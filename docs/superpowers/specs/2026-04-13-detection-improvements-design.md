## Улучшение 1 — Исключить собственный ИНН из клиентских

**Проблема:** ИНН Коловрат/Сидерус (`9701077015`) встречается в 373 из 1790 писем — в цитатах ответных писем, подписях, шаблонах. Система ошибочно принимает его за ИНН клиента.

**Решение:**

В `email-analyzer.js` добавить константу рядом с `OWN_DOMAINS`:
```js
const OWN_INNS = new Set(['9701077015']);
```

При любом извлечении ИНН добавить фильтр:
```js
function isOwnInn(inn) {
  return OWN_INNS.has(String(inn));
}
```

Применить в функциях извлечения ИНН: если найденный ИНН совпадает с `OWN_INNS` — пропустить как кандидат. Логировать в `recognitionDiagnostics` как `own_inn_filtered`.

**Файл:** `email-analyzer.js`  
**Затронутые места:** функция извлечения ИНН из тела письма, из вложений, из подписи (~5 мест)

## Улучшение 2 — Поддержка ИНН ИП (12 цифр) и BY (9 цифр)

**Проблема:** Текущий `INN_PATTERN` поддерживает `\d{10,12}`, но без явной валидации типа. Белорусские ИНН (9 цифр) не попадают в паттерн.

**Текущий паттерн (строка 36):**
```js
const INN_PATTERN = /(?:ИНН|inn)(?:\/КПП)?\s*[:#-]?\s*(\d{10,12})/i;
```

**Изменение:**
```js
const INN_PATTERN = /(?:ИНН|inn|УНП)(?:\/КПП)?\s*[:#-]?\s*(\d{9,12})/i;
```

Добавить функцию классификации ИНН:
```js
function classifyInn(inn) {
  const s = String(inn);
  if (s.length === 9) return 'BY';           // Беларусь (УНП)
  if (s.length === 10) return 'RU_ORG';      // РФ юрлицо
  if (s.length === 12) return 'RU_IP';       // РФ ИП
  return 'UNKNOWN';
}
```

Добавить поле `innType` в результат анализа (`crm.innType`). Для `RU_ORG` сохранить существующую валидацию контрольной суммы, для `RU_IP` и `BY` — только длина.

Добавить `УНП` (белорусский аналог ИНН) как маркер для паттерна.

**Файл:** `email-analyzer.js` строка 36, функция classifyInn новая

## Улучшение 3 — Пропускать ИНН в контексте ЭДО/Диадок/СБИС

**Проблема:** ИНН в подписи иногда принадлежит оператору ЭДО или Диадок, а не клиенту-заказчику. Пример:
```
Документооборот через Диадок, ИНН оператора: 7736207543
```

**Решение:**

Добавить константу:
```js
const EDO_CONTEXT_PATTERN = /(?:диадок|diadoc|сбис|sbis|контур|kontur|оператор\s+эдо|эдо\s+оператор|электронный\s+документооборот|подключен\s+к)\s{0,20}/i;
```

При извлечении ИНН из строки: если строка или предыдущая строка содержит `EDO_CONTEXT_PATTERN` — пометить ИНН как `edo_context` и пропустить как кандидат клиентского ИНН.

Логировать в `recognitionDiagnostics` как `edo_inn_skipped`.

Исключение: если ИНН явно маркирован как `ИНН организации:` или `ИНН клиента:` — доверять этому маркеру даже при наличии ЭДО-контекста в том же блоке.

**Файл:** `email-analyzer.js`

## Улучшение 4 — Полное название должности

**Проблема:** KB-паттерны для должностей имеют приоритеты, но "менеджер" (приоритет 70) может матчиться раньше "менеджера по закупкам и логистике". Результат: в поле `contact.position` попадает просто "Менеджер" вместо полного названия.

**Текущая логика (detection-kb.js строки 120-133):** Паттерны с приоритетами, `matchField("position", body)`.

**Решение:**

В функции `extractPosition` (или в KB matcher) изменить логику: при нескольких матчах в одной строке — выбирать **самый длинный** совпавший текст, а не самый приоритетный. Длинный матч → более специфичная должность.

```js
// Было: выбирать по максимальному priority
// Стало: при равном или близком priority (разница < 20) выбирать более длинный матч
function selectBestPosition(matches) {
  if (!matches.length) return null;
  const maxPriority = Math.max(...matches.map(m => m.priority));
  const candidates = matches.filter(m => maxPriority - m.priority < 20);
  return candidates.sort((a, b) => b.text.length - a.text.length)[0];
}
```

Добавить в KB паттерн для захвата свободного текста должности из формальных блоков:
```js
// "Должность: Инженер АСУТП и КИПиА" → "Инженер АСУТП и КИПиА"
{ pattern: /должность\s*[:#-]\s*(.{3,60}?)(?:\n|$)/i, priority: 95 }
```

**Файлы:** `email-analyzer.js` (функция выбора), `detection-kb.js` (новый паттерн)

## Улучшение 5 — КПП при дедупликации клиентов

**Проблема:** Крупные компании с филиальной сетью имеют один ИНН, но разные КПП для каждого филиала. Сейчас дедупликация по ИНН объединяет все филиалы в одного клиента, теряя информацию о конкретном подразделении.

**Решение:**

В `crm-matcher.js` (и в `email-analyzer.js` при формировании `crm` объекта) изменить ключ дедупликации:

```js
// Было:
const clientKey = inn;

// Стало:
const clientKey = kpp ? `${inn}/${kpp}` : inn;
```

В поле `crm` результата анализа добавить флаг:
```js
crm.deduplicationKey = kpp ? `${inn}/${kpp}` : inn;
crm.isFilialByKpp = Boolean(inn && kpp); // true = возможно филиал
```

При поиске в `company_directory`: сначала искать по связке `inn+kpp`, при отсутствии — откатываться к поиску только по `inn`. Это позволяет знать головную компанию даже если конкретный филиал не найден.

**Файлы:** `email-analyzer.js`, `src/services/crm-matcher.js`

## Улучшение 6 — Ограничить бренды из вложений поставщика

**Проблема:** Если письмо классифицировано как «Поставщик услуг» (vendor), его вложения — каталоги поставщика. Система читает из них все бренды, добавляя 50+ посторонних в результат клиентского запроса.

**Текущая логика:** Уже есть лимит 200 строк и cap 15 брендов (commit 5424510).

**Решение:**

Добавить условие в логику обработки вложений: если `classification.label === 'Поставщик услуг'` — **не извлекать бренды из вложений вообще** (только артикулы, если есть явное ценовое предложение).

```js
const skipAttachmentBrands = classification?.label === 'Поставщик услуг';

if (!skipAttachmentBrands) {
  // существующая логика извлечения брендов из вложений
}
```

Для классификации «Клиент» с вложением-спецификацией (несколько брендов + артикулы + количества) — оставить текущий лимит 15 брендов.

Для классификации «Клиент» с вложением — одним брендом (PDF каталога производителя): извлекать бренд + артикулы, игнорировать прочие бренды из документа.

**Файл:** `email-analyzer.js` — секция обработки вложений

## Улучшение 7 — Расширить парсер веб-форм siderus.ru

**Проблема:** На siderus.ru несколько вариантов форм (с загрузкой файла / без, мобильная, страница товара). Текущий парсер ловит часть вариантов.

**Текущие паттерны (строки ~4092-4138):**
- `Заполнена форма|web-форма|Имя посетителя:|Новый заказ|Заказ звонка`
- Поля: Имя, Email, Телефон, Название организации, ИНН

**Расширение:**

Добавить альтернативные заголовки полей из других форм siderus:
```js
const ROBOT_FORM_PATTERNS = {
  name: /(?:Имя\s+посетителя|Имя|ФИО|Контактное\s+лицо)\s*[:\-]?\s*(.+)/i,
  email: /(?:E-?mail|Почта|Электронная\s+почта)\s*[:\-]?\s*([\w.+%-]+@[\w.-]+\.[a-z]{2,})/i,
  phone: /(?:Телефон|Тел|WhatsApp|Мобильный)\s*[:\-]?\s*([+\d][\d\s\-()\/.]{6,})/i,
  company: /(?:Название\s+организации|Компания|Организация|Предприятие)\s*[:\-]?\s*(.+)/i,
  inn: /(?:ИНН)\s*[:\-]?\s*(\d{9,12})/i,
  product: /(?:Товар|Название\s+товара|Запрос|Артикул|Наименование)\s*[:\-]?\s*(.+)/i,
  comment: /(?:Комментарий|Сообщение|Текст\s+заявки|Вопрос)\s*[:\-]?\s*(.+)/i,
  quantity: /(?:Количество|Кол-во)\s*[:\-]?\s*(\d[\d\s,.]*)\s*([а-яёa-z]+)?/i,
};
```

Добавить детектирование формы с файлом: если тело содержит `robot@` отправителя И в письме есть вложение → письмо может быть формой с загрузкой файла, не сбрасывать `webFormSource`.

Добавить обработку формы "Запрос КП" (форма с товарными позициями):
```js
// "Запрошено КП на товары:" или "Список товаров:" → парсить как lineItems
const KP_FORM_PATTERN = /(?:запрошено\s+кп|список\s+товаров|перечень\s+позиций)\s*[:\n]/i;
```

**Файл:** `email-analyzer.js` — функция парсинга robot-формы (~строки 4090-4140)

## Улучшение 8 — Автоимпорт каталога брендов при старте

**Проблема:** После каждого передеплоя на Railway база знаний создаётся с нуля — только 662 дефолтных алиаса. Полный каталог (15 454 алиасов) нужно импортировать вручную через API. Это приводило к 27.7% детекции брендов вместо 81.4%.

**Решение:**

В `server.js` при старте сервера добавить проверку и автоимпорт:

```js
async function ensureBrandCatalogLoaded(detectionKb) {
  const stats = await detectionKb.getStats();
  const aliasCount = stats.brandAliasCount || 0;
  
  if (aliasCount < 5000) {
    console.log(`[startup] Brand aliases: ${aliasCount} < 5000, importing catalog...`);
    const catalogPath = new URL('../data/brand-catalog.json', import.meta.url);
    try {
      const catalog = JSON.parse(fs.readFileSync(catalogPath));
      await detectionKb.importBrandCatalog(catalog.brands || catalog);
      console.log(`[startup] Brand catalog imported successfully`);
    } catch (e) {
      console.error(`[startup] Failed to import brand catalog:`, e.message);
    }
  }
}
```

Вызывать после инициализации `detectionKb` при старте:
```js
await ensureBrandCatalogLoaded(detectionKb);
```

Порог `5000` алиасов — достаточно выше дефолтного (~662), достаточно ниже полного (~15 454), чтобы не ломать кастомные базы.

**Файлы:** `src/server.js`, возможно `src/services/detection-kb.js` (метод `getStats`)

## Критерии готовности

- [ ] ИНН `9701077015` не появляется как кандидат клиентского ИНН ни в одном письме
- [ ] ИНН из 12 цифр (ИП) корректно извлекается и помечается `innType: RU_IP`
- [ ] ИНН из 9 цифр (BY) корректно извлекается и помечается `innType: BY`
- [ ] ИНН рядом с "Диадок"/"СБИС"/"оператор ЭДО" пропускается
- [ ] "Менеджер по закупкам и логистике" → `position: "Менеджер по закупкам и логистике"`, не "Менеджер"
- [ ] Два письма с одинаковым ИНН, но разными КПП → разные ключи дедупликации
- [ ] Письмо "Поставщик услуг" → бренды из вложений не добавляются в результат
- [ ] Форма с файлом с siderus.ru → письмо классифицируется как Клиент, поля извлекаются
- [ ] После передеплоя на Railway каталог брендов загружается автоматически (15 000+ алиасов)
