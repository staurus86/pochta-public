## Блок 1 — Вложения: Authentication required

### Проблема
`handleApi()` содержит глобальный gate `requireAuth(req)` (строка ~572). Браузер при открытии вложения в новой вкладке не отправляет заголовок `Authorization: Bearer`, поэтому всегда получает 401.

### Решение
1. Маршрут `/api/attachments/:key/:file` переносится **до** глобального `requireAuth(req)` в `handleApi`.
2. Внутри маршрута — опциональная авторизация: сначала `extractAuthUser(req)` (Bearer), затем fallback на `url.searchParams.get("token")` через `managerAuth.verifyToken()`.
3. Frontend при рендере ссылок на вложения добавляет `?token=<jwt>` к URL. JWT уже хранится в localStorage/state.

### Интерфейс
```
GET /api/attachments/:messageKey/:filename?token=<jwt>
```
Ответ: бинарный файл с правильным Content-Type. При невалидном токене — 401.

## Блок 2 — Дедупликация артикулов, названий товаров и брендов

### Проблема
Строка `А99L-0159-0409#SSET Novotec Ultra-Clean — 30 шт.` порождает 3 артикула, 2 названия товара, ложный бренд. Корень: несколько паттернов извлекают перекрывающиеся фрагменты; `unique()` делает только exact-string dedup.

### Решение — единая функция `deduplicateByAbsorption(items, mode)`

```
mode: 'keep-longest'  — для артикулов и описаний товаров
mode: 'keep-shortest' — для брендов (длинный = ошибочный захват)
```

**Правила поглощения:**
1. Если строка A является подстрокой строки B → удалить A (`keep-longest`)
2. Если строка A совпадает с первым сегментом B (до `-` / `.`) → удалить A
3. Если токен соответствует `[CamelWord]-[CamelWord]` без цифр (торговое наименование) → добавить в `ARTICLE_NEGATIVE_PATTERNS`

**Три точки применения:**

| Список | После какой строки | mode |
|--------|--------------------|------|
| `allArticles` (~1278) | сборка из всех паттернов | `keep-longest` |
| `lineItems[].description` | `buildLineItems` dedup по ключу | `keep-longest` |
| `detectedBrands` | после `uniqueBrands()` | `keep-shortest` |

**Dedup lineItems по артикулу:** если два items имеют совпадающий артикул после нормализации — объединить, взять наиболее длинное `description` и суммировать количество если оба числовые.

## Блок 3 — Должности: расширенные паттерны

### Проблема
Три конкретных провала:
- `"С уважением, юрист\nКаратун Оксана"` — должность до имени, одно слово
- `"Procurement manager of ITER PPTF Project\nLLC"` — латинская многословная должность
- `"Менеджер отдела продаж\nООО КОЛОВРАТ"` — обрезается до первого слова

### Решение

**А. Единый источник истины для должностных слов:**
Объединить `POSITION_KEYWORDS` и `POSITION_SIGNATURE_PATTERN` — паттерн подписи строится **на основе** `POSITION_KEYWORDS`, не параллельно с ним.

**Б. Новый шаг в `extractPosition` — "должность перед именем":**
```
Паттерн: (?:С уважением|Best regards|Regards|Спасибо|Благодарю)[,.\s]*\r?\n
         \s*(<ДОЛЖНОСТЬ_СТРОКА>)\r?\n
         \s*(<ИМЯ>)
```
Где `<ДОЛЖНОСТЬ_СТРОКА>` — строка начинающаяся со слова из `POSITION_KEYWORDS`, длиной до 80 символов.

**В. Латинские многословные должности:**
```
Паттерн: ^([A-Za-z][A-Za-z\s\-,./]{10,120})$
Условия: не содержит @, не URL, не страна/город из стоп-листа,
         предыдущая или следующая строка — имя или телефон
```

**Г. Полная строка должности:**
В `cleanup()` для поля `position` — не обрезать по первой запятой. Должность обрезается только по `\n` (конец строки).

## Блок 4 — Фильтр собственных реквизитов компании

### Проблема
В Re: письмах телефоны, ИНН, название компании Сайдерус/Коловрат из цитируемой подписи попадают в поля отправителя.

### Константа `OWN_COMPANY_IDENTITY`

```js
const OWN_COMPANY_IDENTITY = {
  phones: ["+7 (499) 647-47-07", "+7 (800) 777-47-07"],
  inn: ["9701077015"],
  kpp: ["773101001"],
  ogrn: ["1177746518740"],
  domains: ["siderus.ru", "siderus.su", "siderus.online", "klvrt.ru"],
  nameParts: ["сайдерус", "siderus", "коловрат", "kolovrat"],
};
```

### Функция `isOwnCompanyData(field, value)`

| field | Проверка |
|-------|----------|
| `phone` | нормализованный номер входит в `OWN_COMPANY_IDENTITY.phones` |
| `inn` | точное совпадение |
| `kpp` | точное совпадение |
| `ogrn` | точное совпадение |
| `email` | домен входит в `OWN_COMPANY_IDENTITY.domains` |
| `company` | содержит слово из `OWN_COMPANY_IDENTITY.nameParts` (case-insensitive) |

### 5 точек применения (возвращают `null` если `isOwnCompanyData = true`)
1. `extractSender` — телефон, ИНН, компания после извлечения
2. `splitPhones` — при нормализации каждого номера
3. `extractCompanyName` — финальная проверка результата
4. `parseRobotFormBody` — поля `phone`, `inn`, `company`
5. `mergeQuotedSenderFallback` — перед мержем данных из цитаты

## Блок 5 — Форм-заявка: полное название товара

### Проблема
`"Название товара: AT 051 DA F04 N 11 DS Пневмопривод"` → извлекается только `AT 051`.

Причина: `product` из формы передаётся в `extractArticles()` которая берёт первый артикул-токен и отбрасывает текстовый хвост.

### Решение

**А. `parseRobotFormBody` возвращает `productFullName`:**
```js
return {
  ...существующие поля,
  productFullName: product,   // полная строка "AT 051 DA F04 N 11 DS Пневмопривод"
  product: extractFirstArticleToken(product),  // только "AT 051 DA F04 N 11 DS" (артикул)
}
```

**Б. Новая функция `extractFirstArticleToken(text)`:**
Ищет первый токен в строке соответствующий `ARTICLE_POSITIVE_PATTERNS`. Остальное — текстовое описание.

**В. Pipeline form → lineItems:**
Когда есть `productFullName`, создать lineItem:
```js
{ article: product, description: productFullName, quantity: formQuantity }
```

**Г. Расширение имён полей формы:**
```
Название товара | Наименование товара | Наименование | Товар | Продукт |
Запрос | Модель | Артикул товара | Позиция | Наим\. товара
```

## Блок 6 — Мультибренд: ложный флаг из тела

### Проблема
`multi_brand` ставится когда `detectedBrands.length >= 2` вне зависимости от контекста. Бренды из каталожного перечисления в теле письма дают ложный флаг.

### Решение — источник-взвешенный мультибренд

**Функция `classifyBrandSignal(brand, sourceText)` → `'strong' | 'weak'`:**

`strong` если выполняется хотя бы одно:
- бренд стоит в строке с артикулом (`ARTICLE_CONTEXT_POSITIVE_PATTERNS`)
- рядом есть количество (`qty`, `шт`, `кол-во`)
- бренд в теме письма (`subject`)

`weak` если:
- бренд в предложении с фразой-исключением (каталожный контекст)
- бренд только в теле письма, без артикула рядом

**Фразы-исключения (каталожный контекст):**
```
"также работаем", "можем предложить", "есть в наличии",
"поставляем", "в том числе", "широкий ассортимент",
"официальный дилер", "дистрибьютор", "представитель"
```

**Новое правило `multi_brand`:**
```
multi_brand = (strong >= 2)
           || (strong >= 1 && weak >= 1 && разные артикулы для каждого бренда)
```

Если все бренды `weak` → тип запроса не `multi_brand`, бренды остаются в `detectedBrands`.

## Блок 7 — Флаг массового запроса по CC/To

### Проблема
Если письмо отправлено нескольким получателям — это тендерный обзвон. Менеджер не видит этого и обрабатывает как персональный запрос.

### Решение

**А. Python runner (`project 3/mailbox_file_runner.py`):**
Передавать в `PROJECT3_JSON` для каждого сообщения:
```json
"cc": ["addr1@domain.com", "addr2@domain.com"],
"to_recipients": ["addr1@domain.com"]
```
Собственные адреса (`@siderus.ru`, `@klvrt.ru`) фильтруются.

**Б. `analyzeEmail` принимает новые поля:**
```js
// EmailAnalysisInput
cc?: string[]
toRecipients?: string[]
```

**В. Правило срабатывания флага `mass_request`:**
```
mass_request = true если:
  внешних адресов в CC + To >= 2
  ИЛИ CC содержит >= 1 адрес с доменом отличным от домена отправителя
```

**Г. Флаг в результате:**
Добавляется в `intakeFlow.flags: ['mass_request']`.
В `recognitionIssues` — тип `info`, не влияет на `ready_for_crm`.
Снижает приоритет автосинхронизации в CRM (поле `syncPriority: 'low'`).
